# CONVENTIONS
> Generated: 2026-06-17 | Focus: quality | Project: skills

## Summary
This repo is a skill distribution system, not a traditional software codebase. Conventions govern SKILL.md frontmatter structure, CHANGELOG format, git tag naming, shell script style, and the per-skill versioning model. There is no application code — the primary artifacts are markdown skill definitions and bash packaging scripts.

## Skill File Conventions

**Required frontmatter fields** in `skills/<name>/SKILL.md`:
```yaml
---
name: skill-name                   # kebab-case, matches directory name
description: <long description>    # Used for skill discovery; up to 1024 chars
color: cyan                        # Optional UI color
metadata:
  author: <handle>
  version: X.Y.Z                   # Semver — SINGLE SOURCE OF TRUTH for releases
  description_claudeai: <short>    # Optional; <= 200 chars for Claude.ai packaging
---
```

**`name` field:** kebab-case, must match the directory name under `skills/`.

**`description` field:** Full-length (up to 1024 chars). Describes trigger conditions explicitly. Written in imperative form — "Use this skill whenever..." and "Trigger it even if...".

**`metadata.description_claudeai`:** Optional companion description <= 200 chars for Claude.ai packaging. If absent, `package-skill-claudeai.sh` truncates `description` with a warning.

## Versioning

Each skill is versioned independently using Semantic Versioning (`X.Y.Z`):
- **PATCH** (x.y.Z): bug fixes
- **MINOR** (x.Y.z): new features
- **MAJOR** (X.y.z): breaking changes

Version lives in `metadata.version` in the skill's `SKILL.md` frontmatter. This is the only version field — do not maintain version in any other file.

When changing a skill's `SKILL.md`, always bump `metadata.version` in the same commit (minor bump: `1.1.1` → `1.2.0`).

## Git Commit Message Conventions

**Format:** `<type>: <description>`

**Types observed:**
- `chore:` — tooling, CI, CHANGELOG promotion (e.g., `chore: promote playing-card-prompt CHANGELOG for v3.19.0`)
- `<skill-name>:` — skill-specific changes prefixed with skill name (e.g., `playing-card-prompt: add Joker card support (v3.20.0)`)
- `fix:` — bug fixes
- `Merge ...` — merge commits (standard git format)

**CI auto-commit format:** `chore: promote <skill> CHANGELOG for v<version>` (generated by `package-skills.yml`).

## Git Tag Convention

Tags are namespaced per skill: `<skill-name>/v<version>`

Examples:
- `playing-card-prompt/v3.20.0`
- `content-writer-linkedin/v1.0.0`

## CHANGELOG Conventions

**Two separate changelogs:**
- `CHANGELOG.md` (root) — repo tooling only (scripts, CI, layout). Never skill content.
- `skills/<name>/CHANGELOG.md` — that skill's history only.

**Format:** Keep a Changelog style with `## [Unreleased]` section. On release, CI promotes `## [Unreleased]` → `## [X.Y.Z] - YYYY-MM-DD` automatically.

**Entry style:** Grouped under `### Added`, `### Changed`, `### Fixed`. Entries are detailed — describe what changed and why, including which files were modified.

## Directory and File Naming

**Skill source:** `skills/<skill-name>/` — kebab-case directory
**Assets:** `skills/<name>/assets/<category>/<item>.md` — kebab-case filenames
**References:** `skills/<name>/references/UPPERCASE.md` — uppercase filenames for reference docs
**Scripts:** `scripts/<verb>-<noun>.sh` — kebab-case bash scripts

**File names by role:**
- `SKILL.md` — skill definition (uppercase, required)
- `CHANGELOG.md` — version history (uppercase)
- `README.md` — user documentation (uppercase)
- `TODO.md` — open work items (uppercase)
- `ROADMAP.md` — future plans (uppercase)
- `config.json` — runtime config (lowercase, gitignored if user-modified)

## Shell Script Style

Scripts in `scripts/` follow these conventions:
- Start with `#!/bin/bash`
- Use `set -e` for early exit on errors
- Block comment header explaining usage at top
- Guard clauses with inline error messages: `[ -z "$VAR" ] && echo "..." && exit 1`
- Use `SCREAMING_SNAKE_CASE` for all variable names
- Derive paths from `BASH_SOURCE[0]` rather than assuming cwd

Example from `scripts/package-skill.sh`:
```bash
REPO_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
```

## Skill Content Style

SKILL.md body uses markdown with:
- H1 (`#`) for the skill title
- H2 (`##`) for major sections
- Tables for option/trigger mapping
- Bold for key terms and modes
- Code blocks for exact strings and commands

Trigger conditions are written explicitly and exhaustively in the `description` frontmatter field so the skill fires correctly on natural-language inputs.
